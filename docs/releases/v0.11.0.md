# v0.11.0 - Online Multiplayer MVP

Release date: 2026-05-28

## Summary

`v0.11.0` introduces the first online multiplayer MVP while preserving the existing local play mode. The release adds online room setup, member synchronization, game state persistence, Realtime updates, remote storyteller actions, and visibility filtering for player-facing views.

## Scope

### Local and Online Entry Points

- Local play remains available through the existing local game flow.
- Online play has its own entry pages under `/online`.
- Local storage behavior for local games is preserved.

### Online Rooms

- Create online room.
- Join online room by room code.
- Display room code, room status, and member list.
- Realtime member list updates.
- Add placeholder players for testing low-player-count rooms.
- Delete placeholder players before the game starts.
- Allow ordinary real players to leave waiting rooms.
- Automatically redirect waiting room members when the storyteller starts the game.

### Online Game Start

- Storyteller can start an online game from a waiting room.
- Initial `Game` is written to `game_states`.
- `online_rooms.status` changes to `playing`.
- `online_rooms.game_id` is set.
- `room_members.player_id` is bound to the matching `Game.players[].id` for all participating members.

### Online Game State Sync

- `/online/games/[gameId]` reads game state from Supabase.
- `game_states` Realtime `UPDATE` events trigger client refresh.
- Clients refetch state instead of trusting raw Realtime payloads.

### Remote Actions

Storyteller actions are submitted through:

```text
POST /api/online/games/[gameId]/actions
```

The route reuses existing local game logic:

```text
validateGameAction(game, action)
reduceGameAction(game, action)
```

Supported online storyteller actions in this MVP:

- `ADVANCE_PHASE`
- `SET_DAY_SUB_PHASE`
- `ADD_MANUAL_NOTE`
- `UPDATE_PLAYER_STATUS`
- `RESOLVE_NOMINATION`
- `USE_DAY_ABILITY` for Slayer only
- `APPLY_NIGHT_ACTION` as a generic storyteller night action form

### Action Persistence

Remote action writes are backed by the `submit_game_action` SQL function in `supabase/schema.sql`.

The RPC atomically:

- checks the current `game_states.version`
- inserts a `game_actions` audit row
- updates `game_states.state_json`
- increments `game_states.version`

This avoids the earlier non-transactional risk where `game_actions` could be inserted without the matching `game_states` update.

### Visibility

Online game reads now use `src/lib/visibility.ts`.

Viewer behavior:

- Storyteller receives the full storyteller game state.
- Player receives a filtered game state.
- Spectator receives a public/safe game state.

Ordinary players and spectators do not receive:

- `setupState`
- `nightActionState`
- `history`
- other players' roles
- other players' alignments
- drunk/poisoned state
- storyteller-only logs
- other players' private information

## Supabase Requirements

Apply `supabase/schema.sql` in the Supabase SQL editor before using online mode.

Required environment variables:

```text
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=
SUPABASE_SERVICE_ROLE_KEY=
```

Realtime replication should be enabled for:

- `room_members`
- `online_rooms`
- `game_states`

## Known Limitations

- No Supabase Auth integration yet; temporary user/member identity is stored in browser `localStorage`.
- No player-submitted actions yet.
- No full role-specific online UI yet.
- Night action UI is intentionally generic and storyteller-driven.
- Room ownership transfer, storyteller leaving, and room deletion are not implemented.
- `remoteGameStore` remains a shell for future integration.
